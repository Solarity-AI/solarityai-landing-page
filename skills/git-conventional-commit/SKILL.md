# Git Conventional Commit

Use this skill to run strict conventional commits with controlled staging and
single-line messages.

## Parse Inputs

Accept these optional flags from the user request:

- `--branch <target-branch>`
- `--push`
- `--max-files <n>` (default `25`)
- `--max-lines <n>` (default `800`)
- `--interactive` (prefer `git add -p`)

When a flag is missing, apply defaults.

## Use Git Only

Use only these Git command families for inspection, staging, commit, and push:

- `git status --porcelain=v1`
- `git diff --stat`
- `git diff`
- `git diff --staged --stat`
- `git diff --staged`
- `git add <path>`
- `git add -p`
- `git commit -m "<type>: <subject>"`
- `git rev-parse --abbrev-ref HEAD`
- `git push -u origin main <branch>`

Do not use non-Git command substitutes for this workflow.

## Workflow

1. Inspect current working tree and staged changes with:
   - `git status --porcelain=v1`
   - `git diff --stat`
   - `git diff`
   - `git diff --staged --stat`
   - `git diff --staged`
2. Decide whether to split:
   - compare changed files to `max-files`
   - compare total changed lines (insertions + deletions, approximate) to
     `max-lines`
   - split when concerns are mixed (module/domain/type boundaries)
3. Stage only controlled subsets per commit:
   - never stage all files in one shot on large change sets (`git add .`,
     `git add -A`, `git add --all`)
   - stage by path (`git add path/to/module`) or by hunk (`git add -p`)
4. Review staged content before each commit:
   - `git diff --staged --stat`
   - `git diff --staged`
5. Commit each staged subset with a single-line conventional commit message.
6. Repeat until all intended subsets are committed.
7. If push is requested:
   - use `--branch` when provided
   - otherwise resolve branch via `git rev-parse --abbrev-ref HEAD`
   - run exactly `git push -u origin main <branch>`

## Commit Message Rules

Use this one-line format:

`<type>(optional-scope): <imperative subject>`

Enforce all rules:

- keep one line only
- do not add body text (no second `-m`)
- do not use emojis
- do not include phrases like `generated by ...`
- keep concise (target <= 72 chars)
- use imperative mood (`add`, `fix`, `remove`, `refactor`)

Use one of these types:

- `feat`
- `fix`
- `docs`
- `style`
- `refactor`
- `perf`
- `test`
- `chore`
- `ci`

## Split Ordering Guidance

When commits span multiple concerns, stage and commit in this order:

1. `chore` / `ci`
2. `refactor`
3. `feat`
4. `fix`
5. `docs` / `test`

If one file contains multiple concerns, use `git add -p`.

## Output Requirement

When asked to produce commit message text, output only the single-line
conventional commit message per commit with no extra commentary.
